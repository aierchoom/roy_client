# SecretRoy 分布式系统质量迭代建议

| 项目 | 内容 |
|---|---|
| 文档类型 | 质量改进建议 / 技术迭代方案 |
| 适用对象 | 架构、客户端、服务端、同步协议维护者 |
| 范围 | `roy_client`、`roy_server` 当前本地优先与同步实现 |
| 结论日期 | 2026-04-21 |

## 1. 结论摘要

如果把 SecretRoy 视为一个本地优先的分布式同步系统，而不是企业管理平台，那么当前最值得投入的方向不是组织后台、运营控制台或权限面板，而是把以下几件事做扎实：

- 同步协议的正确性
- 冲突处理的可收敛性
- 本地存储和远端状态的可恢复性
- 身份与加密边界的真实性
- 状态机、错误分类和调试能力
- 自动化测试的覆盖深度

它现在已经有一个值得继续打磨的骨架：

- 本地 SQLite 为主存储，网络同步是后置协调。
- 已经引入 `serverVersion`、`syncStatus`、`HLC`、`Conflict Inbox` 等关键概念。
- 客户端和服务端都保持了相对简单的同步协议，便于继续收敛。

但它仍然明显是“原型级分布式实现”，还没有到“高可信本地优先系统”的程度。主要问题不在界面是否够像企业软件，而在于系统不变量还不够清晰，协议边界和恢复语义还不够硬。

## 2. 应该如何重新定义目标

建议把这个项目的目标定义为：

**一个以本地为真源、以最终收敛为目标、以冲突可解释和恢复可验证为原则的分布式凭证库。**

按这个目标看，评估重点应从“管理能力”切换成这六项：

| 维度 | 当前判断 |
|---|---|
| 本地优先语义 | 有基础，但边界还不够严格 |
| 同步协议正确性 | 有雏形，但冲突分支覆盖不足 |
| 收敛性 | 局部可行，尚未形成可证明的不变量 |
| 崩溃恢复 | 有部分保护，但链路还不完整 |
| 身份与安全真实性 | 仍是原型实现 |
| 可验证性 | 测试样本太少，难支撑后续演进 |

## 3. 当前最关键的问题

### 3.1 系统身份还不是真实的协议身份

`roy_client/lib/services/identity_service.dart` 里最需要先修的不是 UI，而是身份语义本身：

- `vaultId` getter 直接返回固定值 `pub_test_global_vault_001`。
- 私钥和对称密钥仍是 mock 数据。

这会直接伤害分布式系统最基础的隔离性和可推理性：

- 不同设备是否真的属于同一个 vault，当前语义不可靠。
- 多 vault、多测试环境、多设备并行时，元数据很容易互相污染。
- 后续很难严肃地讨论“设备身份”“消息来源”“签名验证”“快照归属”等问题。

结论：

- 先把 identity model 做真，比先做大而全的后台设计更重要。

### 3.2 加密和签名边界目前还是占位实现

`roy_client/lib/sync/sync_service.dart` 中：

- `_encryptAndSign()` 只是把 JSON 做 `base64Encode`
- `_decryptAndVerify()` 只是 `base64Decode`

这意味着当前同步层没有真正的：

- 机密性
- 完整性保护
- 来源可验证性
- 篡改检测

这不仅是“安全问题”，也是“分布式正确性问题”。因为没有真实签名和认证，就无法把“谁写入了这个版本”作为协议事实来依赖。

结论：

- 同步 payload 的真实性必须尽快落地，否则很多冲突和恢复路径都缺少可信前提。

### 3.3 冲突恢复只覆盖了很窄的一条分支

`roy_client/lib/sync/sync_service.dart` 的 `_handleConflict()` 当前只处理：

- `itemId != null`
- 且 `serverActual == 0`

也就是“远端记录缺失”这一类情况。

但真正的分布式冲突不止这一种：

- 本地基于旧版本修改，远端已前进
- 两端都修改了不同字段
- 两端都删除/恢复同一记录
- 本地 tombstone 与远端更新交错
- 模板结构变化导致数据字段解释发生偏移

当前行为更像：

- 只对一种冲突给出补偿动作
- 其余冲突最多重试 3 次后失败

这会导致系统具备“能发现冲突”，但还不具备“能稳定消化冲突”的能力。

结论：

- 下一阶段最重要的不是把 conflict inbox 做得更漂亮，而是先补完冲突分类和恢复路径。

### 3.4 CRDT 合并已有思路，但不变量还需要正式化

`roy_client/lib/sync/crdt_merge_engine.dart` 已经做了几件很有价值的事：

- tombstone 优先
- 字段级 HLC 比较
- 冲突日志保留被覆盖值
- 合并后根据结果决定 `synchronized / pendingPush / conflict`

这条路线是对的，但现在还缺几样关键东西：

- 没有一份正式列出的 merge invariants
- 没有 property-based test 去验证收敛性
- 对模板变化、字段删除、未知字段兼容的讨论还不够
- `serverVersion` 与字段级 HLC 的组合语义还不够明确

建议把下面这些不变量文档化并测试化：

- 相同输入必得相同输出
- 合并顺序变化后最终应收敛
- 删除与恢复的胜负规则应稳定
- 冲突日志不能影响最终收敛
- 任一 merge 不能制造非法状态

### 3.5 同步元数据的命名空间还不彻底

`SyncService.initialize()` 里：

- `sync_version_$vaultId`
- `sync_last_time_$vaultId`

是按 vault 隔离的。

但 `dirty` 标记当前还是：

- `sync_dirty`

这意味着如果未来同一设备上支持多个 vault，或者切换身份/测试环境，dirty 状态就可能串用。

这类问题现在看起来小，但它本质上是在破坏“本地元数据属于哪个副本”的边界。

结论：

- 所有同步元数据都应该以 vault 为命名空间隔离。

### 3.6 本地持久化有恢复意识，但还没形成完整恢复链

`roy_client/lib/services/secure_storage_service.dart` 里已经有一些好迹象：

- 数据库打开失败时尝试备份坏库
- `replaceDatabase()` 采用 temp + backup 的原子替换思路
- schema 已包含冲突日志和同步元数据

但从分布式恢复角度看，还差三块：

- 缺少“本地库损坏 -> 重建 -> 重新拉取 -> 验证完成”的闭环
- 缺少“部分拉取成功但版本号未推进”的幂等验证
- 缺少“冲突日志、本地 tombstone、同步状态”在恢复后的完整一致性检查

结论：

- 现在更像“局部有保护”，还不是“系统级可恢复”。

### 3.7 服务端协议简单，但还不够稳固

`roy_server/index.js` 的优点是简单、容易理解、行为可预期，但它的问题也很明确：

- 持久化介质是单文件 JSON
- 协议依赖单调 `currentVersion`
- 服务端不理解业务语义，只做版本门卫
- 没有真正的快照/日志分层

这不是坏事，反而说明它非常适合继续迭代成“薄同步协调器”。但要达到更高质量，还需要明确两件事：

1. 这个后端到底是“当前状态存储”还是“操作日志协调器”
2. 当前版本号模型要不要继续承担全部一致性职责

如果继续走薄后端路线，我更建议：

- 服务端继续只做版本协调和耐久写入
- 客户端负责加密、签名、字段级冲突解释
- 但服务端需要更明确的持久化语义、错误语义和恢复语义

## 4. 当前系统最值得保留的部分

这套系统并不是要推倒重来，反而有几块很值得保留：

1. 本地优先架构方向是对的

- SQLite 在客户端，网络只是同步层，这一点很适合凭证库产品。

2. `HLC + serverVersion + conflict inbox` 的组合有潜力

- 它兼顾了本地编辑体验、冲突解释和服务器协调成本。

3. 薄后端思路是合理的

- 对这类产品来说，把大部分语义放在客户端而不是服务端，不是缺点，前提是协议边界要更严格。

4. 存储 schema 已经开始为冲突和删除建模

- `is_deleted`、`delete_hlc`、`server_version`、`sync_status` 这些字段让后续演进有抓手。

## 5. 下一阶段应该重点迭代什么

### 5.1 第一优先级：把协议事实做真

这是最核心的一层，建议优先顺序如下：

1. 真正实现 vault / device identity
2. 真正实现 payload encryption + signature
3. 把同步元数据彻底 namespaced
4. 把错误码和冲突类型做成正式协议语义

目标不是上更多功能，而是让系统里的“版本”“作者”“来源”“归属”都变成可信事实。

### 5.2 第二优先级：把冲突路径补齐

建议给冲突做正式分类，而不是只依赖一个 409：

- stale base version
- remote missing
- concurrent edit
- concurrent delete
- resurrect after delete
- invalid payload
- identity mismatch

然后为每类冲突明确：

- 自动处理策略
- 是否进入 inbox
- 是否允许继续 push
- 是否需要用户确认
- 是否需要重新 pull snapshot

### 5.3 第三优先级：把恢复链闭环

建议补足三类恢复演练：

- 本地数据库损坏恢复
- 远端快照重建恢复
- 中途崩溃后的重复同步恢复

一个分布式系统真正的质量，不在于“正常路径多顺”，而在于“坏路径还能不能稳稳收回来”。

### 5.4 第四优先级：把测试从样例测试升级为系统测试

当前测试远远不够支撑这套同步模型持续演进。

建议补四层测试：

- 单元测试：HLC、merge、version gate、storage migration
- 场景测试：双设备编辑、双设备删除、断网恢复、重复 push
- 性质测试：交换顺序、重复合并、最终收敛、不变量不破坏
- 故障测试：脏库、损坏快照、半成功提交、冲突重试

特别建议增加：

- model-based sync tests
- merge invariants tests
- crash recovery tests

## 6. 推荐的分阶段路线

### Phase 0：两周内

目标：先让系统更可推理。

- 修复文档和 UI 中的乱码
- 写出同步协议与 merge invariants 文档
- 统一同步元数据命名空间
- 补充关键错误分类
- 增加 `sync_service` 和 `crdt_merge_engine` 的回归测试

### Phase 1：一到两个月

目标：让协议从“能跑”升级到“可信”。

- 实现真实 device / vault identity
- 落地真实加密与签名
- 重构 conflict handling，覆盖主要冲突分支
- 明确 tombstone、restore、field merge 的最终语义
- 补足崩溃恢复和重复同步测试

### Phase 2：两到三个月

目标：让系统从原型同步走向稳定同步。

- 梳理服务端持久化语义
- 明确 snapshot 与 version 的演进策略
- 增加协议级调试信息与结构化错误
- 建立端到端多设备同步测试基线
- 建立数据修复与重建工具

## 7. 最值得立刻开工的任务

如果现在只做 7 件事，我建议是：

1. 去掉 `vaultId` 固定返回，建立真实 vault 身份
2. 把 `_encryptAndSign()` 从 base64 占位改成真实实现
3. 给 `sync_dirty` 按 vault 进行命名空间隔离
4. 扩充 `_handleConflict()`，至少覆盖 stale base version 和 concurrent edit
5. 给 `CrdtMergeEngine.merge()` 补正式不变量测试
6. 做一组双设备并发编辑/删除/恢复的集成测试
7. 写一份“崩溃恢复与重新同步”操作文档

## 8. 一句话判断

SecretRoy 现在最需要的不是“更像企业软件”，而是“更像一个可信的分布式系统”。只要把身份、协议、冲突、恢复和测试这五层补硬，它的质量会比单纯做界面升级有价值得多。

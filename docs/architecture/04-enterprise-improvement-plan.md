# SecretRoy 企业级改进建议

| 项目 | 内容 |
|---|---|
| 文档类型 | 改进建议 / 整改方案 |
| 适用对象 | 产品负责人、设计负责人、技术负责人、项目决策人 |
| 范围 | `roy_client`、`roy_server`、`docs` 当前实现 |
| 结论日期 | 2026-04-21 |

## 1. 结论摘要

SecretRoy 当前更接近“有方向感的产品原型”和“本地优先密码库技术验证”，还不能称为企业级产品。

它的优点很明确：

- 客户端分层已经有雏形。
- 本地优先和同步冲突处理有一定思路。
- Flutter 桌面端和移动端共用一套 UI 基础，后续统一体验的成本不高。

但短板也很集中：

- 安全能力仍处于原型级，离企业级密码管理器要求差距很大。
- 服务端还只是开发型同步后端，不具备正式生产治理能力。
- 产品信息架构、文案一致性、异常状态设计、企业管理能力都不成熟。
- 工程质量和运维能力不足，测试、审计、监控、备份、恢复都不完整。

如果要把这个项目往“企业级”推进，建议不要只做 UI 美化，而是按“产品定位 -> 安全底座 -> 平台能力 -> 企业体验 -> 运维治理”这条线整体升级。

## 2. 当前成熟度判断

| 维度 | 评级 | 判断 |
|---|---:|---|
| 产品完成度 | 2/5 | 具备基础功能，但缺少企业场景必须的管理与协作能力 |
| 视觉与交互 | 2.5/5 | 有统一风格尝试，但信息架构、文案、状态反馈仍不够专业 |
| 安全能力 | 1.5/5 | 方向正确，但当前实现不能支撑企业级安全承诺 |
| 后端能力 | 1.5/5 | 仅适合开发/演示，不适合正式部署 |
| 工程质量 | 2/5 | 有模块化意识，但测试覆盖和职责边界不足 |
| 运维与治理 | 1/5 | 缺少监控、审计、备份恢复、发布治理闭环 |

## 3. 关键问题清单

### 3.1 产品与设计层面

1. 产品定位不够清晰

- 当前命名、界面语义和文档容易让人误以为这是成熟安全产品，但代码实现仍是原型级。
- 对企业用户来说，最先看的不是“能不能存密码”，而是“能不能管人、管权限、管风险、管审计”，这些核心能力目前缺位。

2. 信息架构偏“功能堆叠”，不够“业务导向”

- `Settings` 承载了外观、安全、同步、模板、版本说明等大量入口，但缺少“企业管理”“组织安全策略”“成员与权限”“审计与告警”等面向企业的一级导航。
- 账户、模板、同步、冲突处理彼此之间的关系对新用户并不直观。

3. 文案和本地化体系不统一

- 项目同时存在 `AppLocalizations`、页面内 `_text()` 双语分支、硬编码英文/中文等多套方式。
- 多个界面和文档已经出现乱码/编码异常，说明内容资产管理和编码链路不稳定。

4. 缺少企业产品常见的状态设计

- 空状态、加载状态、同步异常、权限失败、冲突待处理、风险提醒等状态大多停留在 Snackbar 或简单提示层。
- 缺少“下一步建议”和“管理员可执行动作”，界面专业感不足。

5. 缺少“企业可感知价值”的关键功能

- 没有组织空间、团队共享、审批、离职交接、敏感条目托管、批量导入导出、策略管控、设备可信管理等能力。

### 3.2 安全与合规层面

1. 主密码管理仍是原型实现

- `roy_client/lib/services/enhanced_crypto_service.dart` 当前是把主密码读出后直接比对，并非正式的密钥派生、密钥管理和零知识式设计。
- 这意味着当前实现不能支撑“企业级安全”对外表达。

2. 身份体系仍是 mock/过渡态

- `roy_client/lib/services/identity_service.dart` 中 `vaultId` getter 直接返回固定值 `pub_test_global_vault_001`。
- 同文件中的私钥、对称密钥也仍是 mock 概念，不是真实可信身份体系。

3. 同步安全边界不完整

- 当前同步模型已有版本控制和冲突恢复思路，但整体仍偏开发态。
- 服务端 `roy_server/index.js` 只做基础参数校验和 JSON 文件读写，不具备正式认证、授权、租户隔离、密钥治理和审计追踪能力。

4. 存在高风险“便捷模式”

- `roy_client/lib/views/unlock_view.dart` 与 `roy_client/lib/services/service_manager.dart` 存在 `no password mode` 逻辑。
- 这个能力在内部调试可以存在，但如果缺少严格隔离，极易破坏产品的安全心智。

5. 缺少企业合规能力

- 没有审计日志。
- 没有管理员可见的安全事件。
- 没有策略执行记录。
- 没有权限变更追踪。
- 没有备份、恢复、密钥轮换、异常访问告警等机制。

### 3.3 后端与平台层面

1. 服务端不是正式后端，只是轻量同步协调器

- `roy_server/index.js` 当前使用本地 JSON 文件作为 vault 持久化介质。
- 这种实现适合开发验证，不适合企业环境中的并发、恢复、审计和扩展要求。

2. 缺少正式的账户和组织模型

- 没有用户、组织、工作区、成员、角色、设备、令牌、会话等核心实体。
- 也没有 SSO、SCIM、邀请、禁用、离职回收等企业必需能力。

3. 缺少部署与环境治理

- 目前没有看到正式的环境配置分层、配置中心、密钥注入、数据库迁移、灰度发布、回滚策略等能力。

4. 可观测性不足

- 健康检查很基础，但没有结构化日志、指标、链路跟踪、错误分类、告警规则。
- 出了问题后难以快速判断是客户端、网络、同步协议还是存储损坏导致。

### 3.4 架构与工程层面

1. `ServiceManager` 责任过重

- `roy_client/lib/services/service_manager.dart` 同时负责解锁、自动锁定、同步、存储、身份、密码工具、应用生命周期等职责。
- 这类“总控对象”在原型期很方便，但会随着功能增长迅速膨胀，后续难测、难拆、难治理。

2. Provider 与服务边界仍然偏紧

- `EnhancedAppProvider` 同时承担列表状态、筛选、刷新、同步联动、冲突计数等职责。
- 未来如果加入团队库、共享库、管理员视图，会进一步复杂化。

3. 测试覆盖明显不足

- 客户端当前只有 `test/sync/crdt_merge_engine_test.dart`。
- 服务端当前只有 `test/index.test.js`。
- 对密码管理器这类高风险产品来说，当前覆盖度远不足以支撑快速演进。

4. 文档资产质量参差不齐

- `docs/README.md`、`docs/architecture/00-executive-summary.md`、`docs/architecture/03-risks-and-roadmap.md` 以及若干 UI 文件已经出现乱码。
- 这会直接伤害团队协作、外部评审和交付专业度。

## 4. 直接证据

| 现象 | 证据位置 | 说明 |
|---|---|---|
| 主密码直接比对 | `roy_client/lib/services/enhanced_crypto_service.dart` | 当前不是正式 KDF + 数据密钥方案 |
| 固定 vault 标识 | `roy_client/lib/services/identity_service.dart` | 身份体系仍是过渡态 |
| 服务端使用本地 JSON 持久化 | `roy_server/index.js` | 不具备企业级后端治理能力 |

| 设置页承担过多入口 | `roy_client/lib/views/settings_view.dart` | 信息架构偏消费级工具，不像企业控制台 |
| 多处文案乱码 | `roy_client/lib/views/settings_view.dart`、`roy_client/lib/views/sync_settings_view.dart`、`docs/README.md` | 编码与文案资产管理存在问题 |
| 测试样本极少 | `roy_client/test/sync/crdt_merge_engine_test.dart`、`roy_server/test/index.test.js` | 高风险模块缺少系统性验证 |

## 5. 企业级改造方向

### 5.1 先修“底座”，不要先修“皮肤”

企业级不是把界面做深色、做高级卡片、加几组图表就够了。这个项目最先要补的是以下底座：

- 正式的身份与组织模型
- 正式的安全与密钥体系
- 正式的服务端持久化和认证授权
- 正式的审计、监控、备份和恢复
- 正式的测试与发布门禁

### 5.2 设计要从“工具界面”升级为“企业控制台”

建议把产品结构从“账户列表 + 设置中心”升级为更适合企业的软件结构：

- 总览 Dashboard：风险、同步状态、成员状态、策略执行情况
- Vault 管理：个人库、团队库、共享库、敏感条目
- 成员与权限：成员、角色、组、设备、邀请、禁用
- 安全策略：密码策略、MFA、会话时长、导出限制、审批规则
- 审计与告警：访问记录、导出记录、风险事件、异常设备
- 系统设置：品牌、集成、同步节点、备份恢复、版本信息

### 5.3 体验要从“能用”升级到“可信、可控、可解释”

建议重点提升：

- 明确的风险提示，而不是仅弹出提示条
- 可追踪的同步流程，而不是只告诉用户成功/失败
- 更好的表单质量，包括校验、草稿、撤销、批量编辑
- 可理解的冲突处理，给出原因、影响和建议动作
- 面向管理员的策略配置和执行结果反馈

## 6. 分阶段改进路线

### Phase 0：两周内的快速修正

目标：先把“明显不专业”的问题降下来。

- 修复全部乱码和编码问题，统一文案源。
- 统一中英文策略，禁止页面内继续散落硬编码文案。
- 清理会误导用户的命名，把“原型能力”和“正式能力”明确区分。
- 梳理导航结构，把“设置大杂烩”拆成更清晰的一级入口。
- 为同步失败、冲突、离线、空状态补齐更正式的界面反馈。
- 增加最基础的 smoke test 和关键回归测试。

### Phase 1：一到两个月的安全底座重建

目标：把“不能称企业级”的核心原因先补上。

- 重做主密码体系，采用正式 KDF、数据密钥、密钥封装方案。
- 移除或严格隔离 `no password mode`，仅保留开发环境开关。
- 建立真实的 vault / user / device 身份模型。
- 明确本地库加密边界与同步 payload 加密边界。
- 为同步请求加入认证、授权、设备信任与会话机制。
- 建立审计日志最小闭环。

### Phase 2：两到三个月的平台能力升级

目标：从“个人工具”升级到“组织可用”。

- 服务端从 JSON 文件迁移到正式数据库。
- 引入组织、成员、角色、共享资源模型。
- 增加管理员后台能力。
- 引入 SSO / MFA / 邀请 / 禁用 / 离职回收。
- 增加备份、恢复、导入导出、版本保留和恢复点。
- 建立配置分层、数据库迁移、环境管理和发布流程。

### Phase 3：三到六个月的企业体验打磨

目标：让产品看起来、用起来、管起来都像企业软件。

- 重做首页为运营和安全总览。
- 增加审计检索、风险筛选、告警中心。
- 补齐批量操作、审批流、共享模板、策略推送。
- 支持更专业的报表与导出。
- 针对桌面端和移动端分别优化使用场景，而不是只做响应式适配。

## 7. 具体整改优先级

### P0：必须先做

- 主密码与密钥体系重构
- 真实身份与 vault 模型
- 服务端认证授权
- 数据持久化从 JSON 迁移到正式数据库
- 去掉误导性的“企业级安全”表达
- 修复乱码与本地化体系
- 增加关键路径自动化测试

### P1：尽快做

- 管理员视角导航与企业控制台结构
- 审计日志与安全事件
- 成员、角色、设备管理
- 备份恢复与导入导出
- 结构化日志、监控、告警

### P2：中期做

- SSO / SCIM / 企业目录集成
- 批量治理与审批流
- 更高级的共享和团队协作能力
- 更完整的合规与报表体系

## 8. 建议的目标产品形态

如果这款产品真要做成企业级，建议目标形态不要再定义为“带同步的密码本”，而是：

**面向团队和组织的本地优先凭证与敏感信息管理平台。**

这个定义更贴近后续该做的事情：

- 不只是存密码，还要管理身份、共享、权限和风险。
- 不只是同步数据，还要可追踪、可治理、可审计。
- 不只是好看，还要让管理员有掌控感，让普通用户有信任感。

## 9. 一句话建议

这个项目值得继续做，但下一阶段最重要的不是“再润色一下界面”，而是先把安全、身份、后端、审计和产品结构重新立起来；否则界面越精致，和真实能力之间的落差反而越大。

# 数据库链路同步机制与架构设计说明书 (v1.1.0)

## 1. 概述
本文档旨在详细阐述 Roy 客户端的分布式数据同步机制。系统采用 **Client-Server-Client** 架构，核心目标是在保证端到端加密（E2EE）的前提下，实现多终端数据的最终一致性（Eventual Consistency）。

---

## 2. 核心架构设计

### 2.1 隔离架构 (Vault-Scoped Isolation)
同步逻辑严格以 **Vault (保险库)** 为单位进行隔离。
- 每个 Vault 拥有独立的同步序列号（Global Version）。
- 切换 Vault 时，同步状态、偏移量（Local Version）和冲突队列会自动切换，确保跨保险库数据绝不交叉。

### 2.2 数据模型：CRDT 与 HLC
为解决分布式系统中的并发冲突，Roy 采用了以下技术方案：
- **HLC (Hybrid Logical Clock)**: 混合逻辑时钟。结合物理时间与逻辑计数器，为每一次修改提供全局唯一的因果序。格式为 `timestamp:counter:nodeId`。
- **LWW-Element-Set (Last Write Wins)**: 针对字段级别的“最后写入者获胜”策略。
- **字段级穿透合并**: `CrdtMergeEngine` 会对 `AccountItem` 的每个字段（如 `name`, `email`, `data[key]`）进行独立 HLC 比对，实现“缝合式”合并，而非简单的整条记录覆盖。

---

## 3. 同步协议流程

同步采用 **Pull-then-Push** 的重试循环机制，分为三个阶段：

### 第一阶段：拉取与合并 (Pull & Merge)
1. **获取更新**: 客户端发送 `GET /sync?since={local_version}`。
2. **解密验证**: 客户端接收加密 Payload，使用保险库对称密钥解密，并验证 Ed25519 签名。
3. **CRDT 合并**:
   - 若本地数据较旧，执行 **Fast-Forward**（直接覆盖）。
   - 若存在并发修改，执行 **Conflict Merge**：
     - 比对每个字段的 HLC。
     - 将被覆盖的本地旧值存入 `ConflictLog`（冲突收件箱）。
     - 标记记录状态为 `SyncStatus.conflict`，提示用户人工核对。
4. **墓碑处理**: 检查 `deleteHlc`。若远端删除权早于本地最新修改，则执行本地删除；反之，本地修改会“复活”该记录并产生冲突标记。

### 第二阶段：推送 (Push)
1. **收集脏数据**: 扫描本地 `pendingPush` 状态的记录。
2. **乐观并发控制 (OCC)**: 客户端提交时携带 `expected_base_version`。
3. **服务端检查**:
   - 若服务端当前版本与 `expected_base_version` 不符，返回 `409 Conflict`。
   - 若一致，服务端原子化写入并推进全局版本号。

### 第三阶段：冲突恢复与重试 (Recovery Loop)
若 Push 阶段返回 409，客户端会进入 **Conflict Recovery** 状态：
1. 记录 `_SyncRecoveryMarker` 防止同步中断。
2. 触发一轮强制 Pull，将远端领先的数据合并到本地。
3. 重新计算本地修改，并进入下一轮 Push 尝试（最多重试 3 次）。

---

## 4. 安全协议 (Security)

同步链路实现了 **零知识 (Zero-Knowledge)** 架构：
- **端到端加密**: 所有同步 Payload 在客户端使用 AES-256-GCM 加密后再上传。
- **防篡改签名**: 每个 Payload 附带 Ed25519 签名。服务端仅负责存储，无法解密，也无法伪造客户端数据。
- **传输层安全**: 强制要求 HTTPS 或局域网受信链路。

---

## 5. 业务风险评估 (Risk Assessment)

### 5.1 数据一致性风险
- **风险**: 网络中断可能导致同步到一半挂起。
- **对策**: 引入了 `_SyncRecoveryMarker`（恢复锚点）。系统在启动时会检查上一次失败的阶段，并强制执行一次全量 Snapshot 对齐，确保本地版本号与服务端始终保持最终一致。

### 5.2 并发冲突风险
- **风险**: 两个设备同时修改同一账号的同一字段。
- **对策**: LWW 机制确保系统不会死锁或出现分叉。通过 `ConflictLog` 机制将“被输掉”的数据保留在本地，用户可以在“冲突中心”查看并手动找回。

### 5.3 墓碑攻击 (Tombstone Risks)
- **风险**: 恶意或错误设备发送大量删除指令。
- **对策**: 逻辑删除机制。所有的删除操作都带有 HLC。如果一个设备在被删除后又有了更新的修改事实（HLC 更晚），记录会被自动“复活”，有效防御了误删导致的永久丢失。

### 5.4 性能与规模风险
- **风险**: 随着记录增加，Snapshot 可能会过大。
- **对策**: 采用了基于 Version 的增量同步（Since 参数）。只有在首次绑定或发生灾难恢复时才执行全量拉取。

---

## 6. 结论
Roy 的同步机制通过 **HLC 因果序 + 字段级 CRDT 合并 + 强制恢复锚点**，构建了一个极其健壮的去中心化数据网络。该方案在处理弱网、离线编辑和多端并发方面具有较高的技术容错能力，风险处于受控状态。

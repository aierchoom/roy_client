# 账户与模板调查总结报告

**生成日期**: 2026-04-19
**项目**: SecretRoy
**调查范围**: 账户、模板、模板字段、账户编辑、模板编辑、模板删除、账户切模板、同步影响
**调查依据**: 当前 `roy_client` 代码实现

---

## 1. 调查目标

本次调查的目标是回答三个问题：

1. 当前账户与模板的真实数据关系是什么
2. 客户执行模板相关操作后，数据库和业务行为会发生什么
3. 当前整体编辑设计是否足够安全，是否需要调整

---

## 2. 当前实现概况

### 2.1 数据关系

当前系统中：

- `accounts` 表保存账户记录
- `templates` 表保存自定义模板
- 账户字段值存放在 `accounts.data(JSON)`
- 账户只通过 `template_id` 关联模板
- 内置模板写死在客户端代码中，不存在数据库表里

因此，模板与账户不是强外键关系，而是“模板定义 + 账户 JSON 值”的弱关联关系。

### 2.2 当前编辑机制

账户编辑页的工作方式是：

1. 根据当前模板生成字段控件
2. 用账户 `data` 中对应 key 的值填充控件
3. 保存时按当前页面可见控件重新组装新的 `data`

模板编辑页的工作方式是：

1. 编辑模板字段列表
2. 删除字段时只改模板定义
3. 不对历史账户数据做迁移或批量修正

### 2.3 当前保护机制

当前仅有的明显保护是：

- UI 层阻止删除“正在被账户使用的自定义模板”

但这条规则主要写在页面层，不是底层统一业务规则。

---

## 3. 主要调查结论

## 3.1 模板字段变更不会自动改写历史账户

这是当前实现的核心事实。

模板变更后：

- 模板结构立即变化
- 历史账户的 `data(JSON)` 不会同步迁移
- 账户和模板之间可能逐步失配

因此，模板字段变更不是简单的“界面变化”，而是潜在的数据结构变更。

## 3.2 删除模板不会自动删除账户，但会产生高风险孤儿账户

如果模板被强制删除：

- 账户记录仍然存在
- `template_id` 变成悬空引用
- 账户旧字段值短期仍在数据库里
- 列表页和搜索页仍可能看到这些数据
- 但账户编辑页失去模板定义后，后续保存极易破坏原数据

结论：

**模板删除当前不是“安全保留账户”，而是“把账户推进高风险孤儿状态”。**

## 3.3 删除模板字段不会立刻删值，但会引发延迟型数据丢失

删除模板字段后：

- 模板里不再定义这个字段
- 历史账户里原值仍然暂时保留
- 编辑页不再显示该字段
- 后续再次保存账户时，这个旧值大概率被静默清理

结论：

**当前最大风险不是立刻删数据，而是延迟型数据丢失。**

## 3.4 账户切模板属于高风险结构操作，不应被当作普通下拉切换

当前切模板时：

- 页面会按新模板重建字段控件
- 旧模板字段不会自动迁移
- 一旦保存，旧结构里的值容易被覆盖掉

结论：

**账户切模板本质上是迁移行为，不是普通编辑行为。**

## 3.5 当前编辑器设计存在结构性缺陷

当前编辑器最大的问题是：

**模板不仅控制展示，还间接控制数据是否继续被保留。**

也就是说，模板实际上拥有了“数据裁剪权”。

这会导致：

- 只要模板不再显示某字段
- 该字段就很容易在后续保存中被系统抹掉

所以问题不是某个按钮逻辑有 bug，而是编辑设计本身就存在高风险。

---

## 4. 风险汇总

## 4.1 数据风险

- 模板删除导致孤儿账户
- 字段删除导致延迟型丢值
- 切模板导致大面积字段丢失
- 外部修改 `fieldKey` 会直接断开数据映射

## 4.2 产品风险

- 用户看到的数据和真实可维护状态不一致
- `isSearchable`、`isCopyable` 等属性与实际行为不完全一致
- 客户难以预判“这次改模板会影响哪些账户”

## 4.3 架构风险

- 关键业务规则只在 UI 层生效
- 底层删除模板没有统一保护
- 同步仍是整库推拉，结构变更冲突成本高

---

## 5. 最终判断

## 5.1 是否需要调整整体设计

需要。

但不建议彻底推翻当前“模板 + JSON data”的整体方向。

真正需要调整的是：

- 账户编辑器的保存方式
- 模板变更的处理方式
- 模板缺失和字段缺失的产品语义

## 5.2 推荐方向

建议把当前系统从：

**模板决定显示什么，也间接决定保存什么**

改成：

**账户保存完整状态，模板只负责展示、校验和结构约束，危险变更通过迁移流程处理**

---

## 6. 建议原则

后续改造建议遵循以下原则：

1. 任何旧字段默认保留，除非用户明确删除
2. 模板缺失时账户默认只读，禁止直接覆盖保存
3. 删除字段、切模板、改 key 都视为结构迁移
4. 危险规则不能只写在 UI，必须下沉到客户端服务层
5. 先止血，再重构，再升级模板治理能力

---

## 7. 调查结论摘要

一句话总结：

**当前系统最大的问题不是模板改了之后数据库不会自动迁移，而是编辑器会把“模板当前看不见的数据”当成“不需要保留的数据”。**

因此，这次改造的第一优先级不是“先做复杂迁移系统”，而是：

**先让保存行为安全，再逐步引入迁移能力。**

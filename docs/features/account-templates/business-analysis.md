# 账户与模板业务分析

**生成日期**: 2026-04-19
**更新日期**: 2026-05-07
**分析范围**: `roy_client` 当前代码实现下，账户、模板、模板字段变更对数据库和用户行为的影响
**结论原则**: 以代码现状为准，不按理想设计推断

---

## 1. 当前实现概览

### 1.1 数据模型

当前核心关系不是"字段单独存表"，而是：

- `accounts` 表保存账户主数据
  - `template_id`: 账户关联的模板 ID
  - `data`: 一个 JSON 字符串，保存模板字段值，结构类似 `{"username":"alice","password":"***"}`
- `templates` 表保存自定义模板定义
  - `fields`: 一个 JSON 字符串，保存字段列表及属性
- 内置模板不在 `templates` 表中，而是写死在客户端代码里（`lib/models/account_template.dart`）

这意味着：

1. 账户字段值不是按列存储，而是整体塞进 `accounts.data`
2. 模板字段变更时，数据库不会自动逐条改写历史账户
3. 模板和账户之间是"弱关联"，不是数据库外键级联

### 1.2 当前代码里的关键事实

以下结论基于对 `lib/services/secure_storage_service.dart`、`lib/views/templates/template_list_view.dart`、`lib/views/templates/template_edit_view.dart`、`lib/views/accounts/account_edit_view.dart` 等文件的逻辑分析：

- `accounts` 和 `templates` 是两张独立表，`accounts.template_id` 只是普通文本字段。
- 删除模板时，只删除 `templates` 表对应记录，不会处理 `accounts` 表。
- UI 层删除自定义模板前，会先检查是否仍被账户使用；若有使用则阻止删除。
- 已存在字段的 `fieldKey` 在模板编辑页被锁定，避免直接改 key。
- 删除模板字段只是把字段从模板内存列表移除，没有任何账户迁移逻辑。
- 账户编辑页只按"当前模板字段列表"生成编辑控件，再把这些控件重新序列化为新的 `data`；模板不存在时直接返回，不会恢复旧字段；保存时只写当前控件里的 key/value。
- 账户列表页如果发现 `account.data` 里有模板未定义的旧 key，仍会兜底显示这些值。
- 搜索页会直接搜索 `account.data.values`，不依赖模板字段是否还存在。

### 1.3 内置模板现状

当前客户端硬编码了 **4 个内置模板**：

| 模板 ID | 标题 | 分类 | 字段数 | 特殊字段 |
|---|---|---|---|---|
| `builtin_generic_info` | 网站模板 | `login` | 5 | `totp` (custom + isReference) |
| `builtin_secure_note` | 通用安全笔记 | `note` | 1 | `content` (longText + isSecret) |
| `builtin_mnemonic` | 助记词 | `note` | 1 | `mnemonic_words` (list + isSecret) |
| `builtin_api_service` | API 服务 | `note` | 3 | `api_keys` (list + isSecret) |

内置模板字段均携带默认 `Hlc.zero('builtin')`，支持字段级 CRDT 合并。`longText` 和 `list` 类型的值均以普通字符串形式存入 `accounts.data`（`list` 使用 `\n` 分隔），同步协议无需改动。

### 1.4 字段级 HLC 与 CRDT 合并

每个 `AccountField` 现在有 4 个独立的 HLC 时间戳：

- `labelHlc` — 标签名称的修改时间
- `descriptionHlc` — 字段描述的修改时间
- `attributesHlc` — 字段属性（类型、是否必填等）的修改时间
- `orderHlc` — 排序的修改时间

在 `CrdtMergeEngine.mergeTemplate()` 中，模板合并是**逐属性 LWW（Last Write Wins）**：
- 远程 `labelHlc` 更新 → 采用远程标签
- 本地 `attributesHlc` 更新 → 采用本地属性（如字段类型）
- 冲突时生成 `TemplateConflictLog`，记录到 `conflictLogs`

这意味着：
- 多端同时修改同一字段的不同属性（如 A 改标签、B 改类型），可以正确合并。
- 但多端同时修改同一属性的同一字段，仍然会产生冲突。

## 2. 一个重要判断标准

分析模板变更影响时，必须分开看两层：

- **数据库物理层**: 数据有没有立刻被删掉
- **产品可见层**: 用户还能不能继续看到、编辑、保留这些数据

在当前实现里，很多场景都是：

- 数据库里暂时还在
- 但 UI 不再把它当成正式字段
- 用户下一次编辑并保存时，这些旧数据会被静默清理掉

这类情况风险很高，因为它不是"立刻丢失"，而是"延迟丢失"。

## 3. 关键场景分析

## 3.1 客户删除模板

### 当前实现下的真实行为

#### 场景 A: 通过当前 UI 删除"正在被使用的自定义模板"

- 当前 UI 会拦截，不允许删除
- 所以正常点删时，数据库不会变化

#### 场景 B: 模板被绕过 UI 删除

包括但不限于：

- 通过脚本/SQL 直接删库
- 同步覆盖时模板在另一端已被删掉
- 未来若增加批量删除、服务端删除但未做同样校验

这时会发生：

- `templates` 表中该模板记录被删除
- `accounts` 表中原有账户记录还在
- 这些账户的 `template_id` 仍然指向旧模板 ID
- `accounts.data` 原始 JSON 仍然保留

### 数据库影响

| 项 | 影响 |
|---|---|
| `templates` | 删除对应模板记录 |
| `accounts.template_id` | 不变，变成悬空引用 |
| `accounts.data` | 不变，原字段值仍在 |

### 用户可见影响

- 账户列表仍能显示这些账户，但模板会变成"未知模板"
- 账户列表会兜底展示 `account.data` 里剩余的字段
- 搜索页仍可能搜到这些旧值
- 账户编辑页拿不到模板定义，字段区会空掉
- 一旦用户重新保存这个账户，极易把旧 `data` 清空或部分清空

### 风险判断

**高风险**

原因：

1. 模板删掉后，账户没有立即消失，容易让人误以为安全
2. 但编辑页失去字段定义后，下一次保存存在静默数据丢失风险
3. 这是"延迟型数据损坏"，最容易在生产中被忽视

### 建议策略

- 删除模板必须在服务层和存储层再次校验"是否被账户引用"，不能只靠 UI
- 不建议直接物理删除模板，建议改为"归档/停用"
- 若必须删除，应该先弹出影响分析
  - 使用该模板的账户数
  - 删除后账户会进入"待迁移模板"状态
- 模板缺失的账户应进入只读态，禁止直接保存，直到：
  - 恢复模板
  - 重新绑定到新模板
  - 完成字段映射迁移

## 3.2 客户删除模板中的特定字段

### 当前实现下的真实行为

删除字段时，当前代码只改模板定义，不改历史账户数据。

因此：

- 模板 `fields` 中这个字段消失
- 历史账户 `accounts.data[oldFieldKey]` 还在
- 列表页和搜索页还能在部分场景下看到这些旧值
- 账户编辑页不会再渲染这个字段
- 用户下一次保存账户时，这个旧 key 会从新 `data` 里消失

### 数据库影响

| 项 | 影响 |
|---|---|
| `templates.fields` | 删除该字段定义 |
| `accounts.data` | 不会立刻删除旧 key |

### 用户可见影响

- 短期：旧值仍可能在列表/搜索里露出
- 中期：编辑页不再可见，也不能编辑
- 长期：一旦该账户被再次保存，旧字段值通常会被真正移除

### 风险判断

**高风险**

这比"直接删数据"更危险，因为用户很难感知：

1. 数据看起来还在
2. 但编辑时已经不可见
3. 下一次保存才真正丢

### 建议策略

删除字段前必须给出处理选项，而不是直接删定义：

1. 仅从模板隐藏，保留账户历史值
2. 删除模板字段，但把历史值迁移到"归档字段"
3. 删除模板字段，并批量删除所有账户中的该字段值

其中第 3 种必须二次确认，并显示影响账户数。

---

## 3.3 客户新增模板字段

### 当前实现

- 只会更新模板定义
- 历史账户不会自动补值
- 如果新增字段是必填字段，旧账户下次编辑保存时会被拦住，要求补填

### 数据库影响

| 项 | 影响 |
|---|---|
| `templates.fields` | 新增字段定义 |
| 历史 `accounts.data` | 不自动补该 key |

### 风险判断

**中风险**

风险点：

- 新增必填字段后，历史账户会进入"必须补录才能保存"的状态
- 如果是批量很多账户，客户会感受到业务阻塞

### 建议策略

- 为新增字段提供默认值策略
- 对"新增后设为必填"增加提示：会影响多少历史账户

---

## 3.4 客户修改字段名称、说明、排序

### 当前实现

- 这些修改只影响模板展示层
- 历史账户值不会改
- 因为 `fieldKey` 不变，账户数据还能继续映射

### 风险判断

**低风险**

这是当前实现里相对安全的一类变更。

### 例外

- 若客户把字段名称改得与实际含义不一致，会造成业务语义错乱
- 排序变化会改变编辑页录入顺序，可能影响操作效率

---

## 3.5 客户修改字段 `fieldKey`

### 当前实现

- 对已存在字段，UI 已禁止直接改 `fieldKey`
- 这是正确的保护

### 但仍有外部风险

若未来通过脚本、导入、服务端同步、数据库修复等方式改了 `fieldKey`：

- 模板会开始寻找"新 key"
- 历史账户数据还在"旧 key"
- 编辑页会把新字段当成空值
- 旧 key 可能只在列表兜底中短暂可见
- 下次保存账户时，旧 key 大概率丢失

### 风险判断

**高风险**

`fieldKey` 本质上是账户值和模板字段之间的稳定主键，不能当普通名称修改。

### 建议策略

- 继续禁止已落库字段改 key
- 如确需改 key，必须走"字段迁移"流程
  - 旧 key -> 新 key 批量搬迁
  - 校验成功后再切模板定义

---

## 3.6 客户修改字段类型

### 当前实现

字段类型主要影响 UI 表现，不会自动转换历史值。

例如：

- `text -> password`: 旧值仍在，但变成掩码显示
- `text -> time`: 旧值仍在，但可能不符合时间格式
- `text -> number/email/url`: 当前代码没有严格格式迁移和校验
- `text -> longText`: 旧值仍在，UI 从单行变为多行
- `text -> list`: 旧值仍在，但 list 会以整段文本作为第一行展示

### 风险判断

**中风险**

风险不一定是"立即丢数据"，但可能造成：

- 展示异常
- 输入控件不匹配
- 用户误以为数据格式已被系统修正，实际上没有

### 建议策略

- 修改字段类型时显示兼容性提醒
- 对不兼容类型变更提供数据预检查

---

## 3.7 客户修改字段属性

### `isRequired`

- 历史数据不补值
- 空值旧账户在下次保存时会被拦截

风险：**中风险**

### `isEditable`

- 字段会变成只读
- 原值仍在
- 不能继续在编辑页修改

风险：**中风险**

### `isSecret`

- 只影响展示和遮罩
- 原值仍在

风险：**低风险**

### `isSearchable`

- 当前代码里基本没有按该属性控制搜索
- 搜索页仍然会扫描 `account.data.values`

风险：**中风险**

原因不是数据损坏，而是"产品承诺和实际行为不一致"。

### `isCopyable`

- 当前代码里复制按钮仍可用，没有真正按该属性拦截

风险：**中风险**

### `isPrimary`

- 当前实现里更多是元数据，影响有限

风险：**低风险**

---

## 3.8 客户切换账户所属模板

这是一个非常容易被忽略的高风险场景。

### 当前实现

当用户在账户编辑页切换模板时：

1. 当前字段控制器会被清空并按新模板重建
2. 若是编辑旧账户，因为新模板 ID 不等于旧模板 ID，不会自动迁移旧值
3. 保存时只会保存新模板字段对应的数据

### 结果

- 老模板下的原始字段值不会自动迁移
- 如果用户直接保存，旧数据会大面积丢失

### 风险判断

**高风险**

### 建议策略

- 切换模板时必须弹迁移确认，不允许静默切
- 至少提供以下选项：
  1. 保留旧数据为归档字段
  2. 手动映射字段
  3. 明确确认"放弃旧字段值"

---

## 3.9 客户删除账户

### 当前实现

- 只删除 `accounts` 表中的账户记录
- 模板不受影响

### 风险判断

**低风险**

这是当前实现中最清晰的一类删除行为。

---

## 3.10 内置模板变更或删除

内置模板不在 `templates` 表，而在代码里写死。

这意味着：

- 内置模板字段一旦改动，会直接影响所有使用该模板的账户
- 风险模式和自定义模板相同，但影响面更大
- 如果代码层移除某个内置模板，相关账户会直接变成"未知模板账户"

当前 4 个内置模板中，`builtin_generic_info` 是最老的内置模板，使用面最广；3 个 `note` 分类模板是新增模板，目前影响面较小。

### 风险判断

**高风险**

---

## 3.11 多端同步下的额外风险

当前同步是"整库文件推送/拉取 + 版本号比较"，不是按账户、按模板、按字段做细粒度合并。

因此当多端同时做这些操作时：

- A 设备删字段
- B 设备改账户
- C 设备删模板

最终结果不是结构化合并，而可能是整库覆盖。

### 风险判断

**高风险**

### 建议策略

- 在现有同步架构下，模板和字段变更应视为高危险操作
- 至少增加：
  - 同步前后变更摘要
  - 模板删除/字段删除的冲突提示
  - 关键操作的备份点

## 4. 风险总表

| 操作 | 数据库即时影响 | 用户短期感知 | 最终风险 |
|---|---|---|---|
| 删除正在使用的模板（当前 UI） | 无 | 删除被阻止 | 低 |
| 强制删除已使用模板 | 模板记录删除，账户保留 | 账户变未知模板 | 高 |
| 删除模板字段 | 只删模板定义，不删账户旧值 | 短期还能搜到/看到，编辑页消失 | 高 |
| 新增字段 | 只改模板定义 | 历史账户无值 | 中 |
| 新增必填字段 | 只改模板定义 | 旧账户下次保存被拦 | 中 |
| 改字段标题/说明/顺序 | 仅模板元数据变更 | UI 文案变化 | 低 |
| 改字段 key | 外部改动会断关联 | 新字段空、旧值成孤儿 | 高 |
| 改字段类型 | 仅模板定义变更 | UI 表现变化 | 中 |
| 改 `isEditable` | 仅模板定义变更 | 字段变只读 | 中 |
| 改 `isSearchable` | 当前几乎不生效 | 行为与预期不一致 | 中 |
| 改 `isCopyable` | 当前几乎不生效 | 仍可复制 | 中 |
| 切换账户模板 | 保存后重写 `data` | 旧值极易丢失 | 高 |
| 删除账户 | 账户记录删除 | 直接消失 | 低 |

## 5. 最关键的业务风险

## 5.1 延迟型数据丢失

不是删字段那一刻丢，而是下一次编辑保存时丢。

这是当前模板业务里最危险的问题。

## 5.2 UI 保护和底层保护不一致

当前 UI 虽然拦截了"删除正在使用模板"，但底层删除模板接口没有级联校验。

只要入口多一个，风险就会暴露。

## 5.3 模板是弱关联，不是强约束

这让系统很灵活，但也意味着：

- 更容易产生孤儿模板引用
- 更容易出现旧字段残留
- 更需要业务层做迁移和校验

## 5.4 属性配置与实际行为不一致

尤其是：

- `isSearchable`
- `isCopyable`

这类属性已经进入模型，但当前并未形成完整执行闭环。

## 6. 建议的产品规则

建议把模板操作分成三类：

### A. 安全变更

- 改标题
- 改副标题
- 改描述
- 改显示顺序
- 改图标/分类

可直接保存。

### B. 受控变更

- 新增字段
- 改字段类型
- 改是否必填
- 改是否只读

保存前必须做影响提示。

### C. 危险变更

- 删除模板
- 删除字段
- 改字段 key
- 切换账户模板
- 删除内置模板定义

这些必须走迁移流程，不建议直接执行。

---

## 7. 建议的系统改造优先级

### P0

- 账户保存时保留"模板外旧字段"，避免静默清理
- 模板缺失账户进入只读态，禁止直接保存
- 删除模板增加服务层二次校验
- 删除字段前展示影响账户数

### P1

- 模板字段删除支持三种策略
  - 仅隐藏
  - 归档历史值
  - 批量清除历史值
- 账户切换模板时提供字段映射向导
- 对新增必填字段提供默认值或批量补值方案

### P2

- `isSearchable`、`isCopyable` 真正落地
- 为模板和账户增加变更审计日志
- 为高风险模板操作增加自动备份点

## 8. 对你提出的两个问题的直接回答

### 问题 1: 客户删除模板，原本该模板的账户数据会怎么处理？

按当前实现：

- 正常 UI 下，如果模板仍被账户使用，删不掉
- 如果被强制删掉：
  - 账户记录不会自动删除
  - 账户的 `template_id` 会变成悬空引用
  - 账户 `data` 里的原始字段值还在
  - 列表和搜索短期还能看到部分旧值
  - 但编辑页失去模板后，下一次保存非常容易把旧数据清空

所以答案不是"立即删除"或"完全保留"，而是：

**数据库短期保留，业务上进入高风险孤儿状态。**

### 问题 2: 客户删除模板特定字段，原本该模板账户数据的这一项会怎么处理？

按当前实现：

- 模板定义中的字段会消失
- 历史账户的该字段值不会立刻从数据库删除
- 列表/搜索可能暂时还能看到
- 编辑页不会再显示这个字段
- 该账户下一次保存时，这个旧字段值很可能被真正清掉

所以答案是：

**字段值不会立刻物理删除，但会变成"不可维护的旧数据"，后续保存时高概率丢失。**

---

## 9. 最终结论

当前实现下，账户与模板的关系更像"模板定义驱动显示，账户 JSON 自行存值"，而不是严格的关系型结构。

这套设计的优点是灵活，缺点是模板删除、字段删除、字段 key 变更、账户切模板这几类操作都非常容易产生：

- 孤儿数据
- 延迟型数据丢失
- 多端覆盖风险
- 配置语义与实际行为不一致

如果这个产品后续要进入稳定可运营阶段，模板相关危险操作必须从"直接 CRUD"升级为"带影响分析、迁移策略、备份保护的变更流程"。

---

## 10. 整体编辑设计是否需要改变

### 10.1 结论

**需要改，而且建议改的是"编辑架构"，不是只在现有页面上继续打补丁。**

当前这套设计的核心问题不是某一个删除动作没处理好，而是：

- 账户编辑页的真实字段集合，完全由"当前模板定义"临时生成
- 保存时又用"当前页面上存在的字段控件"反向覆盖账户 `data`

这会天然带来一个结构性问题：

**凡是不在当前模板里的字段，都不是一等公民。**

于是就会反复出现：

- 旧字段暂时还在数据库里
- 但编辑器不承认它
- 最终在下一次保存时被系统清掉

所以如果继续沿用现在的设计，只靠增加一些 if/弹窗/校验，风险能下降，但不会真正消失。

### 10.2 当前编辑设计的根本问题

#### 问题 1: 编辑页是"模板投影"，不是"账户真实状态编辑器"

现在的账户编辑页本质上是：

1. 找到模板
2. 按模板字段生成控件
3. 从账户 `data` 里按 key 填充值
4. 保存时只回写这些控件里的值

这意味着编辑器不是在编辑"账户完整状态"，而是在编辑"模板当前允许看见的那一部分状态"。

这是模板强约束场景下最危险的设计点。

#### 问题 2: 模板既是"展示规则"，又是"数据裁剪规则"

理想上模板应该主要负责：

- 字段显示顺序
- 标签名
- 控件类型
- 校验规则
- 分组和展示

但当前实现里，模板实际上还决定：

- 哪些字段会被保留
- 哪些字段会被保存
- 哪些字段会在保存时消失

这就让"UI 模板配置"直接获得了"数据删除权"。

#### 问题 3: 缺少账户级 schema/version/unknown-fields 容器

现在账户只存：

- `template_id`
- `data`

但没有这些关键辅助信息：

- 账户创建时基于哪个模板版本
- 当前账户是否存在模板外字段
- 哪些字段是历史遗留字段
- 哪些字段需要迁移确认

因此系统无法区分：

- 这是合法字段
- 这是旧字段
- 这是待迁移字段
- 这是意外脏数据

#### 问题 4: 模板切换与字段删除没有"迁移步骤"

现在很多危险操作是"改完即生效"：

- 删除字段
- 切换账户模板
- 模板缺失后继续保存

但这些动作本质上都应该是"结构迁移"，不是普通编辑。

---

## 10.3 是否要彻底推翻当前模式

### 不建议彻底推翻的部分

这些设计可以保留：

- 模板驱动账户表单生成
- `accounts.data` 用 JSON 存储动态字段
- 模板定义和账户值分离

这些都很适合低成本支持自定义字段。

### 必须改变的部分

真正要改的是：

1. **保存策略**
2. **孤儿字段处理方式**
3. **模板变更流程**
4. **账户迁移流程**

也就是说，不一定非要换存储模型，但必须把"编辑器如何理解和保存账户状态"改掉。

---

## 10.4 建议的新编辑设计方向

### 方向 A: 账户编辑器必须以"账户完整状态"为准

保存账户时不应只保存"当前模板字段控件"。

应该改成：

1. 先读取账户原始 `data`
2. 再应用当前页面已编辑字段的变更
3. 对模板中不存在但历史上已有的字段，默认保留
4. 只有在用户明确确认删除时，才真正移除这些字段

这会把现在的"覆盖式保存"改成"合并式保存"。

这是最重要的一步。

### 方向 B: 引入"未知字段/历史字段"区域

当账户里存在模板外字段时，编辑页不应该假装它们不存在。

建议增加一个单独区域，例如：

- 历史字段
- 未映射字段
- 待迁移字段

用户至少应该能看到：

- 字段名
- 当前值
- 来源
- 处理方式

处理方式可以是：

- 保留
- 映射到现有字段
- 标记归档
- 删除

### 方向 C: 模板变更要分为"展示变更"和"结构变更"

建议在产品层面把模板编辑拆开：

- 展示变更
  - 改标题
  - 改副标题
  - 改顺序
  - 改说明
- 结构变更
  - 删字段
  - 改 key
  - 改类型
  - 新增必填字段

展示变更可以直接保存。

结构变更必须进入"影响分析 + 迁移策略"流程。

### 方向 D: 账户切模板必须进入迁移向导

切模板不应该是一个普通下拉框立即生效动作。

建议改为：

1. 用户选择新模板
2. 系统展示旧模板字段和新模板字段差异
3. 自动尝试匹配明显同名字段
4. 用户手动处理剩余字段
5. 预览迁移后结果
6. 再确认保存

### 方向 E: 模板缺失账户默认只读

当 `template_id` 找不到模板时：

- 允许查看
- 允许复制
- 不允许直接保存覆盖

除非用户先执行以下任一动作：

- 恢复模板
- 绑定新模板并完成迁移
- 明确删除旧字段

---

## 10.5 如果短期不重构，最少要补哪些点

如果当前阶段不适合做较大改造，建议至少先补这几个底线保护：

### 最小改动 1

账户保存改为：

- `newData = oldData + editedVisibleFields`
- 默认保留旧 key

不要直接 `newData = visibleFieldsOnly`

### 最小改动 2

模板不存在时，账户编辑页禁止保存。

### 最小改动 3

删除模板字段前显示：

- 影响账户数
- 当前策略
- 是否保留历史值

### 最小改动 4

账户切模板时弹确认框，至少先提醒：

- 旧字段不会自动迁移
- 保存后可能丢失原字段值

### 最小改动 5

列表页和详情页明确标记：

- 未知模板
- 历史字段
- 模板外字段

不要让用户误以为这些数据仍处于正常受控状态。

## 10.7 最终判断

如果你的问题是：

"是不是只要补一下删除模板和删除字段的逻辑就够了？"

答案是：

**不够。**

如果你的问题是：

"是不是整套账户编辑设计就需要调整？"

答案是：

**是，需要。**

但不是要彻底推翻"模板 + JSON 字段值"这个方向，而是要把它从"模板决定存什么"改成：

**账户保存完整状态，模板决定如何展示和约束，结构变化通过迁移流程处理。**

## 11. 修改方案

本节给出一个按阶段落地的修改方案，目标不是一步到位重做全部系统，而是：

1. 先停止静默数据丢失
2. 再补齐模板迁移能力
3. 最后把模板系统升级成可运营的结构变更系统

---

## 11.1 方案总览

### 方案目标

要解决的核心问题有四个：

1. 账户保存不能再覆盖掉模板外旧字段
2. 模板删除/字段删除不能再直接制造高风险孤儿数据
3. 账户切模板必须有迁移过程
4. 模板变更要能区分普通编辑和结构变更

### 总体思路

建议采用下面这条主线：

- **存储层**: 继续使用 `accounts.data(JSON)`，但保存策略改为合并式
- **编辑层**: 账户编辑器从"模板投影编辑器"升级为"账户完整状态编辑器"
- **模板层**: 模板继续定义展示与校验，但危险变更进入迁移流程
- **产品层**: 增加"历史字段 / 未映射字段 / 待迁移字段"概念

---

## 11.2 第一阶段方案：先止血

这一阶段尽量不大改表结构，优先把最危险的问题压住。

### 改动 1: 账户保存改为"合并式保存"

#### 当前问题

当前账户保存逻辑相当于：

```text
newData = 当前页面可见字段集合
```

应该改成：

```text
newData = oldData
newData = 用当前编辑值覆盖 oldData 中同 key 的值
只在用户明确删除时移除 key
```

#### 实现建议

在 `account_edit_view.dart` 的 `_save()` 方法附近，把：

- 直接从 `_fieldCtrls` 生成全新 `data`

改成：

- 先复制 `widget.initial?.data ?? {}`
- 再把当前模板字段值逐个写回
- 不在当前模板里的旧 key 默认保留

#### 预期收益

- 删除字段后，打开账户再保存，不会自动丢失旧值
- 模板暂时缺失时，账户原始数据至少还能保住
- 切模板时即使没迁移，也不会第一时间把旧值抹掉

---

### 改动 2: 模板缺失账户默认禁止保存

#### 当前问题

当账户引用的模板不存在时，仍可能进入风险编辑流程。

#### 实现建议

在 `account_edit_view.dart` 中增加判断：

- 如果 `provider.getTemplate(templateId)` 为 `null`
  - 页面进入只读模式
  - 显示"模板缺失，需恢复或迁移后才能保存"
  - 隐藏保存按钮或禁用保存动作

#### 预期收益

- 避免孤儿账户被误保存
- 避免因模板缺失导致二次破坏

---

### 改动 3: 删除模板下沉到客户端服务层校验

#### 当前问题

删除模板的"引用检查"主要在 UI 层。

#### 改造目标

在客户端服务层增加统一规则：

- 查询模板引用账户数
- 被引用时拒绝删除

#### 责任分层

- 页面层：只负责提示和确认
- Provider：负责协调调用
- 服务层：负责执行业务规则
- 存储层：负责具体 SQL

#### 新增建议方法

- `SecureStorageService.countAccountsByTemplate(String templateId)`
- `ServiceManager.deleteTemplateSafely(String templateId)`
- `EnhancedAppProvider.deleteCustomTemplateSafely(String templateId)`

#### 涉及位置

- `template_list_view.dart`
- `service_manager.dart`
- `secure_storage_service.dart`

#### 交付标准

- 无论从哪个入口删模板，只要仍被账户使用，就删除失败

---

### 改动 4: 删除字段前增加影响确认

#### 当前问题

模板编辑页删除字段是直接移除，没有任何影响提示。

#### 改造目标

删除字段前至少展示：

- 字段名称
- 受影响账户数
- 删除后历史值默认保留，但不再属于模板字段

#### 涉及位置

- `template_edit_view.dart`

#### 交付标准

- 用户删除字段前必须经过确认
- 确认文案明确说明风险

---

### 改动 5: 账户切模板增加风险提示

#### 当前问题

切模板当前是普通下拉框选择，风险暴露不足。

#### 改造目标

切模板时增加确认弹窗，说明：

- 旧字段不会自动迁移
- 当前版本先保留旧字段
- 后续会在历史字段区继续处理

#### 涉及位置

- `account_edit_view.dart`

#### 交付标准

- 用户明确确认后才切模板

---

## 12. 第二阶段：编辑器重构方案

目标：让账户编辑器真正编辑"账户完整状态"，而不是只编辑模板当前可见区域。

## 12.1 编辑器状态重构

建议引入统一编辑态模型：

```dart
class AccountEditState {
  final Map<String, String> visibleValues;
  final Map<String, String> legacyValues;
  final Set<String> deletedKeys;
}
```

### 用途

- `visibleValues`：当前模板字段
- `legacyValues`：账户历史字段 / 未映射字段
- `deletedKeys`：用户明确删除的字段

### 好处

- 保存逻辑可控
- 字段删除是显式行为
- 模板外字段不再被隐式吞掉

---

## 12.2 编辑页增加"历史字段区"

### 设计目标

当账户存在模板外字段时，编辑页必须显式展示，而不是隐藏。

### UI 建议

增加一个单独区块：

- 标题：历史字段 / 未映射字段
- 字段内容：key、值、来源
- 可执行操作：保留、删除、映射

### 涉及页面

- `account_edit_view.dart`

### 交付标准

- 模板删字段后，用户能在账户页看见历史字段
- 用户可以明确选择保留或删除

---

## 12.3 列表页风险状态展示

### 设计目标

让用户在进入账户前就知道哪些账户存在结构问题。

### 建议状态

- 未知模板
- 存在历史字段
- 待迁移

### 涉及页面

- `account_list_view.dart`

### 交付标准

- 风险账户在列表中可识别

---

## 13. 第三阶段：模板迁移与治理

目标：让危险模板操作从普通编辑升级为结构变更流程。

## 13.1 引入模板版本

### 建议新增字段

模板增加：

- `version`

账户增加：

- `templateVersion`
- `migrationState`

### 作用

- 判断账户是否滞后于模板
- 判断是否需要迁移确认

---

## 13.2 模板结构变更分类

建议将模板修改分成两类：

### 展示变更

- 改标题
- 改说明
- 改顺序
- 改分类

这类可以直接保存。

### 结构变更

- 删字段
- 改字段 key
- 改字段类型
- 新增必填字段

这类进入迁移流程。

---

## 13.3 账户侧懒迁移

当前阶段推荐采用"懒迁移"：

1. 模板先完成变更
2. 账户在下次打开时检测版本差异
3. 若涉及结构变更，则弹迁移提示
4. 用户确认后再真正调整账户结构

### 选择理由

- 改造成本较低
- 与当前本地 SQLite 架构兼容
- 不需要一次性批量修库

---

## 13.4 切模板迁移向导

中长期建议将切模板从普通下拉改为迁移向导。

### 向导步骤建议

1. 选择新模板
2. 展示旧模板字段
3. 展示新模板字段
4. 自动匹配同名/近似字段
5. 用户手动处理剩余字段
6. 预览保存结果
7. 确认提交

## 16. 验收标准

上线第一阶段后，至少应满足：

1. 删除模板字段后，历史账户再次保存不会静默丢失旧值
2. 模板缺失账户不能直接保存
3. 被引用模板不能从底层直接删掉
4. 切模板前用户能看到风险提示

第二阶段完成后，应再满足：

1. 历史字段在账户编辑页可见
2. 用户可以显式保留或删除历史字段
3. 风险账户在列表页可识别

第三阶段完成后，应再满足：

1. 模板结构变更可追踪
2. 账户可识别是否需要迁移
3. 切模板具备基础映射能力

---

## 17. 推荐决策

建议立即启动第一阶段。

原因很明确：

- 当前已经存在静默数据丢失路径
- 问题不是偶发，而是设计必然产物
- 第一阶段改造成本相对可控
- 能显著降低后续模板运营风险

最终推荐路线是：

**短期止血，中期重构编辑器，长期建立模板迁移机制。**

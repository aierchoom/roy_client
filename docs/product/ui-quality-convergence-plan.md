# 客户端 UI 质量收敛计划

| 项目 | 内容 |
|---|---|
| 状态 | 迭代执行中 |
| 日期 | 2026-04-29 |
| 范围 | 客户端主导航、首页/搜索、账户库、模板管理、设置、同步设置、表单编辑页 |
| 目标 | 将客户端 UI 收敛为更扁平、更清晰、更专业、更适合高频使用的本地优先保险库工具 |

## 背景

近期客户端已经补齐了账户、模板、同步、密钥链接、恢复路线和安全提示等多条业务链路。功能复杂度上升后，UI 需要从偏展示型页面收敛到偏工具型界面：减少装饰层级，突出高频动作，让用户更快完成查找、复制、编辑、同步和恢复判断。

本文档先沉淀当前 UI 观察和迭代候选，不直接代表已经排期的实现承诺。后续进入开发前，应按优先级拆成独立任务并补充验收记录。

## 当前观察

### 已有基础

- 客户端已经具备 `AdaptivePage`、`AdaptiveScaffold`、自适应导航和基础响应式布局，后续不需要重建页面框架。
- 主导航已经形成账户、主页、设置三类入口，整体信息架构可继续复用。
- 账户卡片已经覆盖复制、展开详情、风险标识、密码显隐和快捷编辑等核心操作。
- 同步与密钥恢复链路已经完成一轮业务收口，面对面链接、远程配对、离线恢复码、内部兼容码的语义开始清晰。

### 主要问题

- 视觉语言仍偏展示页：多处存在渐变 Hero、大圆角、阴影、玻璃态容器和装饰性头部。
- 专业视觉体验不足：主题能力主要集中在 `ThemeData` 和用户可选 seed color 上，缺少产品级视觉 token、品牌默认色、状态色规则、统一卡片层级和组件规范。
- 主题表达较散：页面内直接使用 `LinearGradient`、`BoxShadow`、`primaryContainer`、`tertiaryContainer` 等样式组合，导致每个页面局部成立，但整体气质不稳定。
- 页面骨架不统一：不同页面的外边距、最大宽度、列表 padding、分区卡片写法存在差异，容易出现双层留白。
- 同步设置页承载过多：同步控制、密钥恢复、面对面链接、远程配对、离线恢复码、内部兼容码、诊断和技术说明集中在一个长页面里，用户决策成本较高。
- 主页和账户页职责存在重叠：主页偏搜索与快捷入口，账户页偏账户库管理，但导航命名和默认入口还不够直观。
- 账户列表卡片信息密度偏低：单条记录的状态、字段、操作和展开区域较重，移动端一屏可浏览数量有限。
- 模板页偏视觉展示：模板管理本质是配置和维护，但当前卡片、徽标、字段预览和装饰头部占用空间较多。
- 编辑页和设置页存在嵌套卡片倾向：部分表单区域可以更扁平，减少“卡片套卡片”的观感。
- 文案本地化方式混用：当前仍可见 `_text(中文, English)`、`AppLocalizations` 和直接中文字符串并存，后续应逐步统一。

## 设计收敛原则

- 工具优先：账户、模板、同步、恢复都应服务高频操作，不以展示型 Hero 作为主要表达。
- 扁平骨架：页面外层由统一框架负责边距和宽度，内部减少重复 padding、阴影和大圆角容器。
- 信息分层：高频动作常驻，低频说明、危险操作、兼容入口和技术细节收起或进入二级页面。
- 场景分区：同步、面对面链接、远程配对、离线恢复码和内部兼容码应按使用场景区分，不混成一个长列表。
- 移动端优先：标题、徽标、按钮和字段预览不能互相挤压；一屏应尽量看到更多可操作对象。
- 中文优先：新增和重写 UI 文案优先走统一本地化路径，避免继续扩大混用面。

## 风格与主题方向

专业感不等于更复杂的装饰，而是来自稳定、克制、可预测的视觉系统。SecretRoy 的视觉方向建议定位为“可信、安静、高效的本地保险库工具”，避免过度赛博、过度渐变或过度活泼。

### 颜色

- 默认品牌色从通用 `deepPurple` 收敛到更贴近安全工具的低饱和蓝绿或青蓝色，用户仍可在外观设置里选择其他强调色。
- 页面大面积背景使用中性表面色，品牌色只用于主操作、选中态、焦点态和少量关键提示。
- 安全状态使用稳定语义色：成功/可信用绿色，警告/待确认用琥珀色，危险/失败用红色，信息/同步用蓝青色。
- 避免页面大面积由同一色相控制，特别是整页紫色、整页深蓝或整页渐变。

### 字体与层级

- 保留中文友好的 `Noto Sans SC`，但明确标题、列表、标签、说明、按钮的字号和字重层级。
- 页面标题克制，卡片内标题不使用 Hero 级字号。
- 长说明文案降低视觉权重，放在二级区域或折叠区域，不和主操作抢层级。

### 材质与空间

- 普通业务卡片使用低圆角、细边线、少阴影；重要弹窗和底部面板允许更强的容器感。
- 建议建立统一圆角档位：小控件 8，按钮/输入框 10-12，普通卡片 12，弹窗/底部面板 16-20。
- 页面背景、分区、列表项、危险区使用不同层级的表面色，而不是靠大阴影和渐变区分。
- 间距采用固定档位，避免每个页面手写不同 padding。

### 组件规范

- 主按钮、次按钮、危险按钮、图标按钮、选择项、状态 chip、列表行、表单输入、分区标题应有统一样式。
- 账户、模板、同步状态、恢复路线这类核心对象应共享相近的列表密度和状态表达。
- Hero 区域只保留在真正需要建立上下文的页面；常规设置页和工具页应优先使用紧凑分区标题。

## 深层 UI 诊断

当前客户端的视觉问题不是单个页面不好看，而是产品视觉语言没有统一。页面里同时存在三种气质：玻璃拟态导航、渐变 Hero 展示页、工具型数据列表。它们各自局部成立，但组合后会削弱专业感，让应用看起来像不同阶段逐步拼出来的界面。

### 1. 主题层有基础，但还不是设计系统

`main.dart` 已经集中定义了 `ThemeData`、`ColorScheme.fromSeed`、卡片、输入框、按钮和导航主题，这是继续收敛的基础。但当前主题仍偏 Flutter Material 默认能力，缺少 SecretRoy 自己的产品级视觉 token。

主要问题：

- 默认 seed color 偏通用，不足以支撑安全工具的品牌识别。
- 页面背景混入主题色，容易让整套 UI 被单一色相控制。
- 全局卡片、页面卡片、导航容器、弹窗和局部状态块的圆角档位不统一。
- 页面内重复手写渐变、阴影、透明度和边线，导致后续维护成本上升。

收敛方向：

- 建立 `AppRadii`、`AppSpacing`、`AppSurfaces`、`AppTones`、`AppShadows` 这类统一入口。
- 默认主题改为低饱和蓝绿或青蓝色，紫色保留为用户可选主题。
- 新增或重写页面不得继续复制大段 `LinearGradient`、`BoxShadow` 和任意圆角值。

### 2. 导航气质过于悬浮

移动端底部导航和桌面端 dock 当前偏玻璃拟态：大圆角、模糊、阴影和半透明表面较多。这会带来“漂亮壳”的第一印象，但密码管理器更需要稳定、可靠、工具化。

收敛方向：

- 移动端底栏改为更轻的固定导航，不再强调玻璃和大阴影。
- 桌面端使用稳定的侧边 rail 或紧凑 dock，不把主内容包进过重的玻璃容器。
- 导航标签从“主页”决策为“搜索”或“工作台”，让入口语义更贴近日常操作。

### 3. Hero 过度重复

首页、账户页、模板页、设置页、同步页都存在渐变 Hero。结果是每个页面都像首页，用户进入高频工具页面时仍要先跨过一块展示区域。

收敛方向：

- 普通业务页取消大面积渐变 Hero。
- 使用紧凑页面标题、状态摘要、主操作按钮作为页面开头。
- 仅在解锁页、首次引导、重大安全状态页保留更强的 Hero 视觉。

### 4. 模板页偏展示，不像管理工具

模板页当前更像展示卡片集合：渐变头部、贴纸徽标、阴影、字段预览和状态徽标较重。模板本质是配置对象，应优先让用户快速查看字段数、使用数、类型和操作。

收敛方向：

- 自定义模板和内置模板改为管理型列表或轻量网格。
- 模板字段预览保持名称标签，不使用厚边框或多层说明。
- 删除贴纸式徽标，保留更克制的类型标签。

### 5. 同步设置页同时承担太多职责

同步页现在承载同步服务器、即时同步、密钥恢复、面对面链接、远程配对、离线恢复码、内部兼容说明、诊断和技术说明。它是业务风险最高的 UI 页面之一。

收敛方向：

- 拆成“同步控制”“设备链接与恢复”“诊断与技术信息”三类入口。
- 高风险恢复能力必须按场景分区，不和普通同步控制混在同一个长卡片里。
- 内部兼容码继续保持低可见性，不作为普通入口突出展示。

### Step 1. 建立主题与视觉 token

状态：已完成，2026-04-29。

目标：让专业视觉体验有统一来源。

范围：

- 新增或整理统一视觉入口，覆盖颜色语义、圆角、间距、边线、阴影、状态 tone。
- 将默认品牌色从通用紫色调整为更克制的安全工具色。
- 保留外观设置中的用户自定义主题能力。

验收：

- 新增样式可以从统一 token 读取。
- 浅色和深色模式均不依赖大面积高饱和色块。
- 不改变业务流程和页面信息架构。

### Step 2. 收敛页面骨架和普通 Hero

状态：已完成，2026-04-29。

执行边界：本步只收敛普通业务页的页面级 Hero、页面级渐变和主要外边距。模板卡片内部的展示化视觉、贴纸徽标和列表密度问题保留到 Step 5 处理；账户列表密度保留到 Step 6 处理。

目标：先让页面看起来属于同一个产品。

范围：

- 统一 `AdaptivePage` 外边距、最大宽度、列表 padding 和分区间距。
- 设置页、同步页、账户页、模板页的普通 Hero 改为紧凑标题/状态摘要。
- 删除卡片套卡片和重复的页面级渐变。

验收：

- 主要页面左右边距一致。
- 普通业务页不再有大面积渐变 Hero。
- 移动端无文字溢出，桌面端内容宽度稳定。

### Step 3. 拆分同步设置页

状态：已完成，2026-04-29。

执行边界：本步采用页内分区入口，不拆独立路由。保留原有同步、LAN 面对面链接、远程配对、恢复码和诊断方法，只改变同步设置页的信息架构与默认可见内容。

目标：降低同步和恢复入口的误用风险。

范围：

- 将同步控制、设备链接与恢复、诊断与技术信息拆为独立页面或页内分区入口。
- 保持面对面链接、远程配对、离线恢复码、内部兼容码的概念隔离。
- 高风险导入/覆盖动作继续保留明确确认。

验收：

- 用户可以直接进入同步控制或设备链接与恢复。
- 普通同步路径不需要经过恢复说明。
- 现有 LAN、远程配对、恢复码导入流程仍可完成。

### Step 4. 重构导航视觉和入口语义

状态：已完成，2026-04-29。

执行边界：本步只收敛主导航视觉与入口语义。移动端底部导航和桌面侧边导航降低玻璃、阴影和大圆角强度；“主页”入口统一改为“搜索”。模板管理视觉保留到 Step 5，账户列表密度保留到 Step 6。

目标：让导航更像高频工具，而不是悬浮展示容器。

范围：

- 移动端底部导航降低玻璃、阴影和大圆角强度。
- 桌面端侧边导航改为更稳定的 rail/dock，主内容不再被过重容器包裹。
- 决策“主页”是否改为“搜索”或“工作台”。

验收：

- 导航在移动端和桌面端视觉语法一致。
- 主导航标签能直接对应用户意图。
- 键盘弹出、底部安全区和桌面窄宽度不遮挡内容。

### Step 5. 模板页改为管理型界面

状态：已完成，2026-04-29。

执行边界：本步只处理模板列表页的管理型呈现；没有改模板模型、字段编辑器或账户列表密度。模板字段预览改为名称标签，贴纸徽标和重装饰卡片已移除。

目标：让模板维护更直接、更像配置工具。

范围：

- 自定义模板和内置模板改为紧凑管理列表或轻量网格。
- 移除贴纸式徽标和重装饰渐变。
- 字段预览保留名称标签、字段数、使用数和操作入口。

验收：

- 模板标题不被徽标或状态标签挤压。
- 新建、编辑、删除和查看字段更快。
- 内置模板与自定义模板视觉区分明确但不过度装饰。
